## 1. Назначение системы

Система автоматизирует создание и обновление Python-модулей в GitLab с использованием Nexus в качестве репозитория артефактов. При публикации новой версии модуля система самостоятельно создаёт merge request'ы во все зависимые от него проекты с учётом транзитивных зависимостей.

## 2. Архитектура системы

Система состоит из следующих компонентов:

- **GitLab** — хранит репозитории модулей, выполняет CI/CD пайплайны, предоставляет API для управления проектами и merge request'ами.
- **Nexus** — хранит собранные wheel-пакеты, отправляет webhook-уведомления о публикации новых версий.
- **Вебхук-сервис (FastAPI)** — принимает уведомления от Nexus, обрабатывает их и ставит задачи в очередь.
- **Dependency Manager** — строит граф зависимостей между модулями, создаёт ветки и merge request'ы.
- **Скрипты настройки** — автоматизируют первичное развёртывание стенда и создание модулей по конфигурации.

Все компоненты запускаются в Docker-контейнерах и объединены в общую сеть.

## 3. Описание компонентов

### 3.1. Шаблон проекта

Шаблонный проект содержит:
- **`.gitlab-ci.yml`** — пайплайн из двух стадий:
  - сборка wheel-пакета с помощью `uv`
  - публикация пакета в Nexus при создании тега
- **`pyproject.toml`** — базовая структура Python-пакета с версией-заглушкой

### 3.2. Скрипт создания модулей

Скрипт (`project_setup_script.py`) принимает YAML-конфигурацию с описанием модулей и зависимостей, через GitLab API создаёт форки шаблона в заданной группе, удаляет форк-связь и прописывает в каждый модуль его зависимости в формате Git-ссылок на репозитории.

### 3.3. Вебхук-сервис

Реализован на FastAPI. Предоставляет эндпоинты:
- `POST /webhook/nexus` — приём уведомлений от Nexus
- `GET /health` — проверка работоспособности
- `GET /queue-status` — состояние очереди обработки
- `POST /reload-token` — обновление токена GitLab без перезапуска

При получении уведомления сервис проверяет, что событие — создание wheel-файла, извлекает из имени файла название модуля и версию, после чего добавляет задачу в асинхронную очередь. Фоновая задача обрабатывает очередь последовательно, вызывая Dependency Manager.

### 3.4. Dependency Manager

Основные функции:
- Загрузка из GitLab списка всех проектов группы и их зависимостей (парсинг `pyproject.toml`)
- Поиск всех проектов, зависящих от указанного модуля
- Создание ветки в каждом зависимом проекте
- Обновление версии зависимости в `pyproject.toml` (Git-ссылка заменяется на `модуль>=версия`)
- Создание тега и merge request'а

### 3.5. Скрипт настройки стенда

Скрипт (`setup_dev_env.py`) автоматически выполняет первичную настройку:
- ожидает запуска контейнеров
- получает пароль root из GitLab
- создаёт Personal Access Token
- регистрирует GitLab Runner
- настраивает в Nexus репозиторий и webhook
- создаёт группу проектов
- добавляет переменные CI/CD
- создаёт шаблонный проект и загружает в него файлы
- генерирует файл `.env` со всеми параметрами


## 4. Сценарии работы

### 4.1. Создание новой структуры модулей

1. Заполнить `gitlab-scripts/config.yaml`
2. Запустить `python -m src.project_setup_script`
3. В GitLab появляются проекты с настроенными зависимостями

### 4.2. Обновление модуля

1. Разработчик создаёт тег в репозитории модуля (например, `v1.2.3`)
2. CI/CD собирает пакет и публикует его в Nexus
3. Nexus отправляет webhook в вебхук-сервис
4. Вебхук-сервис добавляет задачу в очередь
5. Dependency Manager находит все зависимые проекты, создаёт ветки, обновляет версии и открывает merge request'ы

## 5. Запуск системы

### 5.1. Требования

- Docker и Docker Compose
- Python 3.12

### 5.2. Быстрый старт

```bash
# Запуск контейнеров
docker compose up -d

# Настройка стенда
python setup_dev_env.py

# Создание модулей
cd gitlab-scripts
python -m venv .venv
source .venv/bin/activate
pip install -e .
python -m src.project_setup_script
```
